# V2-00-05 业务编号规范

## 格式

所有公开业务编号统一为：

`{PREFIX}-{RANDOM}`

- `PREFIX` 使用下表固定值。
- `RANDOM` 为 20 位 Crockford Base32 大写随机字符串，排除 `I`、`L`、`O`、`U`。
- 随机部分至少提供 100 bit 熵，不包含数据库自增 ID、日期、用户 ID 或业务数量。
- 展示和搜索时统一大写，允许用户输入小写后标准化。

示例：`GO-ITEM-7K3M9Q2W6R8T4Y5P1HJC`

## 生成规则

1. 由服务端生成，客户端不得提交最终业务编号。
2. 数据库对每个编号列设置唯一约束，并对规范化大写值建立精确索引。
3. 发生极低概率冲突时，事务内重新生成，最多重试三次；仍冲突则失败并告警。
4. 编号创建后永久不变；退款、取消、销毁或匿名化均不复用编号。
5. 业务编号与内部主键分离。内部主键使用 UUID，不在用户界面暴露。

## 搜索和日志

- 列表提供完整编号复制按钮。
- 后台默认只做完整精确搜索；需要前缀搜索的页面必须限制最少 8 个随机字符和返回上限。
- 日志可以记录业务编号，但不得用编号拼接对象存储公开路径。
- 导出保留业务编号，不导出内部 UUID，除非超级管理员执行受审计的技术导出。

## 验收

- 一百万次生成测试无重复且字符集正确。
- 小写输入可标准化精确命中。
- 自增 ID、时间戳、用户 ID 和订单数量无法从编号推断。
- 所有前缀在 API、数据库和 UI 使用同一常量定义。
