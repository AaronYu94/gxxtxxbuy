### 2.1 PRD 明确要求本项由业务拍板

- **附录E（第41页）**：`P0 | 链接解析服务实现方式与支持范围 | 订单开发前`。
- **版本边界声明**：「附录E 中未冻结的参数**不得由研发自行假设为最终业务规则**，应由产品负责人在对应 Epic 启动前确认。」
- **§26 风险表（第13页）** 给出唯一实施指引：「淘宝/1688/Weidian **反爬、登录态和页面变化**可能导致解析中断；建议**先做技术验证；提供人工失败提示和监控**。」

结论：PRD 有意把「用哪个合法数据源、支持哪些平台/字段」留空，交由本决策解决。

### 2.2 工程侧现状（已就绪，等待注入数据源）

V2-03 除本项外的 11 个原子任务已实现并验证（`npm run ci`：lint 141 文件、OpenAPI 85 路径、tests 94/94、build 通过；真实 Postgres 回归通过）。关键：

- **适配器契约已固化**：`fetchProduct(ref) -> 统一快照 | 降级枚举`，任何数据源以 provider 形式注入 `createProductSourceRegistry({ providers })` 即可生效，**无需改动路由、队列、快照或 API**。
- **降级枚举已覆盖**风险表列出的全部失败态：`not_configured / unsupported / item_removed / login_wall / rate_limited / timeout / missing_fields / provider_error`。
- **队列已实现**幂等、指数退避、重试、死信队列 + 告警、超限转人工。
- **人工补录已实现**，且 `scraped` 与 `manual` 数据以 `snapshot.source` 严格区分，快照不可变（数据库触发器强制）。

> 换句话说：**决策批准后，接入任一数据源的工程改动量极小**；决策本身是合规/商业/法律问题，不是技术阻塞。

## 4. 候选方案对比

> 具体服务商名称、真实价格与 SLA 数字**必须由技术/合规负责人尽调后填入**，本表仅给定性量级与结构，禁止把占位数字当事实。

| 维度 | A. 官方/开放平台接口 | B. 授权第三方数据 API | C. 自建解析/采集 | D. 混合（B 为主 + C 兜底 + 人工） | E. 纯人工补录（最小合规） |
| --- | --- | --- | --- | --- | --- |
| 合规性 | 最高（平台授权） | 取决于服务商授权链，需核验 | 高风险（易触 ToS/反爬/法律） | 中（以合规源为主） | 最高（无抓取） |
| 覆盖平台 | 各平台需分别申请，微店开放度低 | 通常三平台都覆盖 | 理论全覆盖，维护重 | 三平台 + 兜底 | 全平台（人工） |
| 字段完整度 | 高但受开放范围限制 | 高（含 SKU/阶梯价/运费） | 取决于自建投入 | 高 | 取决于用户录入 |
| 成功率/稳定性 | 高 | 中–高（受源站波动） | 低–中（反爬/改版易断） | 中–高 | 100%（人工，但慢） |
| 反爬/登录态风险 | 无（走授权） | 由服务商承担 | 高（IP 封禁/登录墙/验证码） | 低 | 无 |
| 接入/上线速度 | 慢（申请/审核周期） | 快（签约即用） | 慢（自研 + 长期维护） | 中 | 最快 |
| 成本量级 | 平台费/分成（待填） | 按量/包月（待填） | 人力 + 代理 IP + 维护（待填） | B 成本 + 少量自建（待填） | 人力成本（随单量线性上升） |
| 维护负担 | 低–中 | 低（外包给服务商） | 高（长期对抗改版） | 中 | 低（但不可规模化） |
| 与现有架构契合 | 直接作为 provider 注入 | 直接作为 provider 注入 | 作为 provider 注入 | 注册表多 provider + 降级链 | 已实现（当前默认） |

### 方案要点

- **A 官方/开放平台**：合规最优。风险是微店开放接口能力有限、申请周期长，可能只覆盖淘宝/1688。
- **B 授权第三方 API**：上线最快、覆盖最全。**合规关键在于核验服务商的数据授权链是否合法**（合规负责人必须审）。
- **C 自建**：控制力最强但法律与反爬风险最高、长期维护成本大；PRD §26 与 D3 均警示「反爬可能增加 2–6 周并需持续维护」。
- **D 混合（工程推荐见 §6）**：合规源为主、自建/人工兜底，成功率与合规兼顾，正好落在 PRD「技术验证 + 人工失败提示 + 监控」的指引上。
- **E 纯人工**：零合规风险、可作为**任何方案的强制兜底**，也是当前 `not_configured` 已实现的状态；缺点是不可规模化。

## 5. 工程推荐（供参考，最终由三方定）

**推荐 D（混合）作为落地形态，分两步：**

1. **首发以合规源为主**：淘宝/1688 优先接入方案 A 或经合规核验的方案 B；微店若无合规源，先走**人工补录（E）**并明确标注。
2. **全平台强制保留人工兜底（E）+ 监控告警**：任一平台解析降级时自动转人工补录，符合 PRD §26 指引。

理由：工程改动量最小（provider 注入即可）、合规风险可控（以授权源为主）、且失败态已被现有降级链与死信/告警完整接住。

> 若三方无法在 `V2-03-01` 前就 A/B/C/D 达成一致，**最低可批准项** = 仅批准 **E（纯人工补录）+ 支持平台范围**，即可解除 03-01 的「无任何合规源」阻塞并进入生产，后续再增补自动源作为新决策版本。
