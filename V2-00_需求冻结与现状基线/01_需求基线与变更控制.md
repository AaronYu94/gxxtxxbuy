## 规则优先级

发生冲突时按以下顺序解释：

1. 已批准的产品变更记录。
2. PRD V2.0 附录 A 的开发级页面与业务规格。
3. PRD V2.0 正文。
4. 本目录中由 PRD 派生的状态、权限、数据和 API 规范。
5. 现有代码行为不构成产品规则；与上述内容冲突时必须修改代码。

## 范围

本次包括账户、链接代购、转运预报、钱包充值、母子订单、采购、到货、QC、仓储、合包、国际物流、优惠券、会员、推广佣金、售后、客服、配置、审计、安全、备份和上线能力。

明确不包括：

- 旧后台数据迁移。
- 传统库存采购和预备库存。
- 独立商品商城、推荐商品或固定商品页。
- 对财务开放利润报表。
- 多级分销。
- 转运商品自动退货。
- 电子产品功能检测。
- 复杂国家级禁运规则引擎。

## 变更记录格式

任何需求变更必须新增一条记录，不允许直接覆盖已批准结论：

| 字段 | 必填规则 |
| --- | --- |
| `change_id` | `GB-CR-` 加 8 位随机大写字母数字 |
| 标题 | 一句话说明变化 |
| 提出人和角色 | 必填 |
| 提出时间 | UTC 秒级时间 |
| 原规则 | 引用 PRD 页码、章节或决策 ID |
| 新规则 | 可直接验收的完整表述 |
| 原因 | 商业、合规、供应商或技术原因 |
| 影响域 | UI、API、数据、状态、权限、资金、测试、运维多选 |
| 影响任务 | 对应 V2 任务 ID |
| 数据兼容 | 无影响、需要迁移、需要双读、禁止变更 |
| 回滚方式 | 必填 |
| 批准人 | 产品负责人；资金规则同时需要财务负责人 |
| 状态 | `draft`、`in_review`、`approved`、`rejected`、`superseded` |

只有 `approved` 状态能改变开发基线。所有批准记录追加保存，不删除、不覆盖。
